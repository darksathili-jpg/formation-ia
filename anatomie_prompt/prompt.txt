### ARCHITECTURE BASE DE DONNÉES (OBLIGATOIRE)

Le TP impose l'utilisation exclusive de SQLite3 pour toute persistance.
Aucun fichier JSON, CSV ou pickle n'est utilisé pour les données
utilisateurs ou les parties.

Schéma de la base `demineur.db` à faire concevoir progressivement
par les élèves :

Table `joueurs` :
  - id               INTEGER PRIMARY KEY AUTOINCREMENT
  - pseudo           TEXT NOT NULL UNIQUE
  - mot_de_passe     TEXT NOT NULL        -- haché avec hashlib.sha256
  - date_inscription TEXT                 -- format ISO 8601

Table `scores` :
  - id           INTEGER PRIMARY KEY AUTOINCREMENT
  - joueur_id    INTEGER REFERENCES joueurs(id)
  - difficulte   TEXT     -- 'facile' | 'normal' | 'expert'
  - duree        INTEGER  -- en secondes
  - mines_total  INTEGER
  - date_partie  TEXT     -- format ISO 8601
  - gagnee       INTEGER  -- 0 ou 1 (booléen SQLite)

Table `parties_sauvegardees` :
  - id              INTEGER PRIMARY KEY AUTOINCREMENT
  - joueur_id       INTEGER REFERENCES joueurs(id)
  - date_sauvegarde TEXT
  - difficulte      TEXT
  - largeur         INTEGER
  - hauteur         INTEGER
  - nb_mines        INTEGER
  - duree_ecoulee   INTEGER  -- secondes jouées avant interruption
  - etat_grille     TEXT     -- sérialisation JSON de la grille
                             -- (seul usage autorisé de JSON :
                             --  comme champ TEXT dans SQLite)

Contrainte : un joueur ne peut avoir qu'une seule partie sauvegardée
active par niveau de difficulté (INSERT OR REPLACE).

Rappels de cours à intégrer dans le TP sur ce thème :
- Notion de base de données relationnelle et de SGBD embarqué
- Différence entre SQLite (fichier local) et un SGBD client-serveur
- Vocabulaire : table, ligne, colonne, clé primaire, clé étrangère,
  contrainte d'intégrité référentielle
- Les quatre opérations CRUD illustrées sur le schéma du TP
- Paramétrage des requêtes (placeholders `?`) pour prévenir les
  injections SQL — point de sécurité obligatoirement mentionné
- Gestionnaire de contexte `with connexion:` pour les transactions

### STRUCTURE DU TP (respecter cet ordre)

**0. En-tête**
- Titre, niveau (Terminale NSI), durée estimée, prérequis listés
- Objectifs pédagogiques formulés en compétences (« être capable de… »)
- Mention : « Ce TP utilise SQLite3, inclus dans Python standard
  sans installation supplémentaire. »

**1. Découverte du jeu — Animation interactive**
- Présentation complète et précise des règles du Démineur
- Animation JavaScript pas-à-pas d'une vraie partie commentée
  (grille 8×8, 10 mines) : le visiteur clique sur « Étape suivante »
  pour avancer ; chaque étape explique le raisonnement du joueur
- Légende visuelle rendue par CSS et texte uniquement, sans emoji :
    - Cellule inconnue : fond gris foncé, texte « ? »
    - Cellule vide révélée : fond clair, contenu vide
    - Cellule chiffrée : fond clair, chiffre coloré selon sa valeur
    - Mine : fond rouge sombre, texte « Mine »
    - Drapeau : fond orange, texte « F »

**2. Modélisation objet — Phase console**

Progression en 7 étapes cumulatives. Chaque étape est structurée ainsi :
  a) Rappel de cours ciblé (encadré « RAPPEL DE COURS »)
  b) Énoncé de la mission, sans jamais fournir la solution
  c) Trois niveaux d'indices masqués par défaut, révélables par un
     bouton « Afficher l'indice » :
       - Indice 1 : orientation générale
       - Indice 2 : piste algorithmique
       - Indice 3 : squelette commenté sans corps de méthode
  d) Éditeur PyScript interactif pré-rempli avec le squelette
  e) Critères de validation précis (sorties attendues)

Étapes :
  1. Classe `Cellule` — état, voisins, méthodes de base
  2. Classe `Grille` — génération aléatoire, calcul des voisins
  3. Classe `GestionnaireDB` — couche d'accès SQLite3 :
       - Méthode `initialiser_base()` : CREATE TABLE IF NOT EXISTS
         pour les trois tables
       - Méthodes CRUD : `inserer_joueur()`, `verifier_joueur()`,
         `sauvegarder_score()`, `sauvegarder_partie()`,
         `charger_partie()`, `supprimer_sauvegarde()`,
         `classement()` (ORDER BY duree ASC WHERE gagnee = 1)
       - Toutes les requêtes paramétrées — aucune f-string SQL
  4. Classe `Joueur` + authentification via `GestionnaireDB` :
       - Inscription : hachage sha256 + INSERT dans `joueurs`
       - Connexion : SELECT + comparaison du hachage
       - Rappel de cours : hachage, mot de passe en clair vs haché
  5. Classe `Partie` — logique de jeu console, récursivité flood-fill :
       - Sérialisation de l'état de la grille en JSON stocké dans
         le champ TEXT de `parties_sauvegardees`
       - Rappel de cours : récursivité, cas de base, pile d'appels
  6. Classe `ScoreManager` — délègue à `GestionnaireDB` :
       - Enregistrement d'un score (INSERT dans `scores`)
       - Affichage du classement (SELECT formaté en tableau console)
  7. Intégration console complète — boucle de jeu, interface textuelle
     soignée, proposition de reprise si une sauvegarde existe

**3. Interface graphique Tkinter**

Progression en 4 étapes :
  1. Classe `VueCellule(tk.Button)` — rendu visuel fidèle au jeu
  2. Classe `VueGrille(tk.Frame)` — placement et événements souris
  3. Classe `Controleur` — patron MVC, liaison Modèle <-> Vue,
     réutilisation de `GestionnaireDB` sans modification
  4. Sauvegarde et reprise de partie :
       - Fermeture : `protocol WM_DELETE_WINDOW` -> dialogue de
         confirmation -> `GestionnaireDB.sauvegarder_partie()`
         avec l'état JSON et la durée écoulée
       - Ouverture : `GestionnaireDB.charger_partie()` -> si une
         sauvegarde existe, proposer « Reprendre » ou « Nouvelle partie »
       - Rappel de cours : événements Tkinter, séparation des
         responsabilités MVC, réutilisabilité de la couche métier

**4. Pour aller plus loin (facultatif — différenciation)**
- Chronomètre persisté dans `scores.duree`
- Grilles personnalisées paramétrables stockées dans la base
- Génération garantissant un premier clic sans mine
- Vue « Statistiques personnelles » : requêtes d'agrégation SQL
  (COUNT, AVG, MIN) présentées comme exercices autonomes
- Index SQL sur `scores(joueur_id, difficulte)` pour optimiser
  les requêtes de classement — introduire la notion d'index
