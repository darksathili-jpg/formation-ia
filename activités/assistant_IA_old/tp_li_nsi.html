1.2 — Caractéristiques techniques100% local · 100% offline · open-source
LLM embarqué — Ollama + llama3.2:3b
Modèle de langage de 3 milliards de paramètres, quantifié en 4-bit (≈2 Go RAM). Génère du texte cohérent localement sur le CPU du RPi5.
~3–8 s par réponse
Transcription vocale — Whisper Small
Modèle de reconnaissance vocale d'OpenAI, exécuté en mode int8 via faster-whisper. Supporte 99 langues.
~1–3 s pour 6 s d'audio
Synthèse vocale — Piper TTS
Moteur TTS neuronal optimisé ARM64. Voix française fr_FR-upmc-medium, naturelle et fluide. Génère un fichier WAV en <1 s.
< 1 s pour une réponse courte
Wake Word — openWakeWord "Hello Li"
Détection du mot déclencheur sans serveur. Modèle ONNX léger (~50 Mo RAM), 3–5% CPU en écoute continue. Entraînable sur vos propres mots.
< 300 ms après le mot
Cœurs CPU alloués : Le RPi5 dispose de 4 cœurs ARM Cortex-A76. Li les répartit ainsi : 1 cœur dédié au wake word + animation, 3 cœurs pour Whisper/Ollama lors de chaque échange.

3.2 — Câblage écran TFT ST7735 → GPIO Raspberry Pi 5Protocole SPI (Serial Peripheral Interface)

Le ST7735 utilise le protocole SPI (Serial Peripheral Interface) : une communication série synchrone entre le RPi5 (maître) et l'écran (esclave). 4 fils suffisent pour transmettre les pixels à 40 MHz.

Broche TFT Pin physique RPi5 GPIO Rôle
VCC Pin 17 3.3V Alimentation écran
GND Pin 20 GND Masse commune
DIN (MOSI) Pin 19 GPIO 10 Données SPI (RPi → écran)
CLK (SCK) Pin 23 GPIO 11 Horloge SPI
CS Pin 24 GPIO 8 (CE0) Chip Select — active l'écran
DC Pin 22 GPIO 25 Data / Command selector
RST Pin 18 GPIO 24 Reset matériel
ST7735 vs ST7789 : Les deux contrôleurs sont compatibles avec le même brochage SPI et les mêmes GPIO. Seule la bibliothèque Python change : adafruit_st7735r pour le ST7735, adafruit_st7789 pour le ST7789. Le code détecte automatiquement lequel est disponible au démarrage.

Détecteur de wake word — li_wake.pyopenWakeWord · Thread d'écoute continue · Callback
Pourquoi 1280 échantillons à 16 kHz ?
1280 / 16 000 = 0.08 s = 80 ms par chunk. openWakeWord analyse des fenêtres de ~80 ms glissantes pour détecter le motif acoustique. Des chunks plus petits augmentent la latence CPU, des chunks plus grands retardent la détection.

5.2 — Architecture orientée objet de LiFaceThread-safe · lock · états

Le threading.Lock() (verrou mutex) empêche les conditions de course : si le thread audio écrit _mouth_open en même temps que le thread face le lit, des valeurs corrompues pourraient apparaître. Le verrou garantit l'accès exclusif à la section critique.
